# Write Log Design

## Purpose

The write log records every planned and applied sync operation. It provides an audit trail, enables replay to reconstruct state, and feeds the sync_metadata hooks on canonical models.

## Log Entry Schema

Each log entry is a JSON object with the following fields:

```json
{
  "operation_id": "uuid4 string — unique per operation",
  "timestamp": "ISO-8601 UTC — when the operation was created",
  "entity_type": "references | tasks | reference_tasks | task_extractions | workflow_states | annotations",
  "entity_id": "canonical id of the affected entity",
  "field": "name of the field being written",
  "old_value": "previous value (null if entity is new)",
  "new_value": "intended value after write (null if entity is removed)",
  "actor": "zotero | notion — which system performed or will perform the write",
  "status": "planned | applied | failed"
}
```

### Field notes

- `operation_id` is generated by the writer at plan time and is stable across retries.
- `timestamp` is set when the entry is first created (plan time), not when it is applied.
- `status` transitions: `planned` -> `applied` on success, `planned` -> `failed` on error.
- `old_value` / `new_value` store JSON-serialisable values. Complex objects (lists, dicts) are stored inline.

## Integration with sync_metadata

Each canonical model carries a `sync_metadata` dict (defined in `core/models.py`). After a successful write the pipeline writes a compact record into that dict:

```python
ref.sync_metadata["last_write"] = {
    "operation_id": op_id,
    "timestamp": ts,
    "actor": "zotero",
    "status": "applied",
}
```

This makes the most recent sync event visible on the entity without querying the log store.

The write log itself is the authoritative record; `sync_metadata` is a cache for quick inspection.

## Storage

- Log entries are appended to a newline-delimited JSON (NDJSON) file per sync run, named `write_log_<run_id>.ndjson`.
- Files are stored in a configurable `logs/` directory (default: `./logs/write_logs/`).
- Each sync run receives a unique `run_id` (uuid4) that groups all entries for that run.

## Retention Policy

- Log files older than 90 days are eligible for deletion.
- A cleanup job (to be scheduled) scans `logs/write_logs/` and removes files whose creation timestamp is older than the retention window.
- Before deletion, entries with `status: failed` should be reviewed or archived separately.

## Replay and Audit

### Reconstructing state

To reconstruct the state of an entity at a given point in time:

1. Collect all log entries for `entity_id` up to the target timestamp, ordered by `timestamp`.
2. Start from the oldest `old_value` of the first `added` entry.
3. Apply each `changed` entry in order, updating the field from `old_value` to `new_value`.
4. The result is the entity state at the target timestamp.

### Audit queries

Common audit operations against the NDJSON log:

- "What changed in the last sync run?" — filter by `run_id`.
- "Who last wrote field X on entity Y?" — filter by `entity_id` and `field`, sort by `timestamp` desc, take first.
- "How many failed operations are there?" — filter by `status: failed`.
- "Show all planned-but-never-applied operations" — filter by `status: planned` from runs older than a threshold.

### Replay

A future `replay` CLI command will accept a `run_id` or `entity_id` and re-execute the planned operations for entries whose status is not `applied`. This is the primary recovery mechanism for partial failures.

## Open Questions

- Compression: NDJSON files can grow large. gzip rotation is a candidate for files older than 7 days.
- Central store: for multi-machine setups, a database backend (SQLite or PostgreSQL) may replace the flat-file approach.
- Sensitive values: `old_value` / `new_value` may contain PII (author names, abstracts). Access to the log directory should be restricted.
